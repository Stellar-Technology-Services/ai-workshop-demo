The codebase: ClaimsChat

  • A real ASP.NET Core / Blazor Server app: C#/.NET, EF Core + SQLite.
  • Browse seeded claim documents, then ask questions about them in a streaming chat with citations.
  • Retrieval and the chat model do the heavy lifting; the app around them is ordinary .NET.
  • That ordinary surface (data, UI, features, tests) is where we work today.

Existing, real, slightly messy. Not a greenfield toy.

Name ASP.NET explicitly (cohort of 15). The retrieval and AI internals are walled off behind interfaces; that comes through naturally when we open the folders, no need to announce it.

Structure a repo for AI

VS Code offers several customization types, each for a different goal. At the end of the day, folders and files.

TypeWhat it doesLives in (VS Code)
InstructionsStandards, conventions, structure: every request, or scoped to files.github/copilot-instructions.md · .github/instructions/*.instructions.md
Prompt filesReusable prompts you invoke as slash commands.github/prompts/*.prompt.md

Full descriptions to say aloud. Instructions: coding standards, conventions, and project structure, applied to every request or scoped to specific files. Prompt files: reusable prompts you invoke as slash commands in chat. Agent skills: multi-step workflows, scripts, and resources the agent loads when a task matches. Custom agents: a specialized persona with its own instructions, tool access, and model. We ship instructions, prompts, and skills in this repo today; custom agents are there when you want a persona. Plan the feature before touching code

/plan-feature

Ticket: let users exclude a document from what the chat can retrieve (soft-delete from retrieval).

  • Decompose: data, seeding, retrieval, UI, tests.
  • Watch the scope balloon: audit log? bulk actions? settings page? Bound it.
  • Be the adult: write down what's out of scope.

Live callback to Session 1's scope-creep lesson. Let the agent propose extras, then visibly cut them. The plan is the contract.

Build it, end to end

  1. Data: add IncludedInRetrieval + an EF migration (hello, SQL).
  2. Seeder: default on insert, but preserve the user's choice on re-seed.
  3. Retrieval seam: a one-line filter, right where the app meets the retrieval internals.
  4. UI: a toggle + an "excluded" badge.
  5. Tests: excluded docs stay out; the flag survives re-seeding.

The seeder step is the planted review beat: a naive impl clobbers the toggle every restart. The seam step is the teachable boundary: we scope data, we don't rewrite the ranking algorithm. Keep your private rehearsal steps handy as a safety net.

Review with a shared standard

/code-review

  • Run the same review skill a senior would, on the diff we just produced.
  • Did it handle re-seeding? Null/empty cases? Did it stay out of the retrieval internals?
  • The standard is the artifact, not one person's memory.

This is the reframed "PR review": survey said PR demos are lowest demand, so we do a shared-standard review of a diff, not a PR walkthrough.

Document for the next dev, and the next agent

  • Capture intent in code docs and comments, not just what the code does.
  • Write down the "why" a diff can't show: the decision, the constraint, the gotcha (like the re-seed trap).
  • The same docs that help a teammate help the next agent read this code. Context compounds.

Dual-audience framing: humans and agents read the same files. We're updating code-level docs as part of this change, not rewriting the repo's AI config files.
